# Database (auth sidecar)

The app uses a MariaDB sidecar for auth-related state (users, login tokens,
sessions, admin tags, sync state). DAV remains the source of truth for contact
data; the sidecar holds what the app writes automatically.

This document covers PR 1: infrastructure only. The schema is added in PR 2.

## Local development

1. Copy `frontend/.env.example` to `frontend/.env` and adjust as needed. The
   defaults match the `docker-compose.yml` setup.

2. Start MariaDB:

   ```sh
   docker compose up -d mariadb
   ```

   Wait for the healthcheck to go green (`docker compose ps`).

3. Apply migrations (no-op in PR 1, will populate in PR 2):

   ```sh
   cd frontend
   npm run db:migrate
   ```

## Environment variables

| Variable      | Default       | Purpose                                |
|---------------|---------------|----------------------------------------|
| `DB_HOST`     | `localhost`   | DB host. `mariadb` inside docker-compose |
| `DB_PORT`     | `3306`        | DB port                                |
| `DB_NAME`     | `jahrweiser`  | Database name                          |
| `DB_USER`     | `jahrweiser`  | Application user                       |
| `DB_PASSWORD` | `jahrweiser`  | Application user's password            |
| `DB_ROOT_PASSWORD` | `rootpw` | Root password (only used by docker-compose to bootstrap the DB) |

Production deployments must override `DB_PASSWORD` and `DB_ROOT_PASSWORD`.

## Schema and migrations

- Schema is defined in `frontend/server/db/schema/` (Drizzle ORM).
- Migrations live in `frontend/server/db/migrations/` and are SQL files
  generated by `drizzle-kit`.

### Generate a new migration after schema changes

```sh
cd frontend
npm run db:generate        # diffs schema vs last snapshot, writes SQL
npm run db:migrate         # applies pending SQL files
```

### Inspect data

```sh
cd frontend
npm run db:studio          # opens drizzle-studio in the browser
```

## Connecting from app code

```ts
import { useDb } from '~~/server/db'

const db = useDb()
// ... drizzle queries here once schema is in place
```

`useDb()` returns a singleton pool-backed Drizzle instance. It reads connection
details from `useRuntimeConfig()`, so it only works inside Nuxt's server
runtime. For CLI scripts, instantiate `mysql2` directly as in
`scripts/db-migrate.ts`.
